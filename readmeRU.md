# 💻 3D Console Engine (C#)

<img width="2070" height="1085" alt="Image" src="https://github.com/user-attachments/assets/06934758-90d4-4b63-9f40-8bb44ca2831e" />

Минималистичный трехмерный движок на C#, работающий полностью внутри системной консоли. Проект построен без использования сторонних графических библиотек и графических API (OpenGL/DirectX). Все вычисления — от кастомной векторной математики до трассировки лучей (Raycasting/Raytracing) и сетевого TCP-стека — выполняются на процессоре (CPU).

Проект создан в демонстрационных и образовательных целях для [YouTube-канала](https://youtu.be/vhYE882B9dE).

---

## 🚀 Основные возможности движка

1. **CPU Raycasting / Raytracing**:
    * Отрисовка полигональных моделей (`.obj`) и геометрических сфер.
    * Оптимизация пересечений через предварительную проверку ограничивающей сферы (Bounding Sphere) перед детальным просчетом треугольников.
2. **Динамическое освещение и тени**:
    * Расчет затухания света (Attenuation) в зависимости от расстояния.
    * Диффузное освещение Ламберта на основе нормалей поверхностей.
    * Реализация теней с помощью пускания теневых лучей (Shadow Rays) из точки пересечения к источнику света.
3. **Многопоточный рендеринг**:
    * Использование `Parallel.For` для параллельного расчета пикселей на все ядра процессора.
4. **Оптимизированный вывод в консоль**:
    * Символьный градиент яркости (`" .:!/r(l1Z4H9W8$@"`) для имитации полутонов.
    * Буферизация кадров и вывод группами одинаковых цветов для минимизации медленных вызовов системного API консоли.
5. **Кастомная 3D-математика**:
    * Собственные структуры `Vector3`, `Vector2`, `Ray`, матрицы поворота (Эйлеровы вращения вокруг осей X, Y, Z).
    * Реализация алгоритма Моллера — Трамбора для пересечения луча с треугольником.
6. **Низкоуровневый ввод**:
    * Опрос клавиатуры в реальном времени через Win32 API (`GetAsyncKeyState`), работающий асинхронно без блокировки потока рендеринга.
7. **Сетевой мультиплеер**:
    * Самописный клиент-серверный TCP-архитектурный модуль.
    * Кастомная бинарная сериализация пакетов, автоматическая маршрутизация через стабильные хэш-идентификаторы типов и система подписок на события.
    * Пример работающего мультиплеерного лобби с синхронизацией позиций игроков и текстовым чатом.

## 🏗️ Архитектура движка

Проект разделен на логические слои, обеспечивающие слабую связанность компонентов:

```
├── AbstractClass             # Абстракции игрового объекта, сцены, экрана и освещения
├── Implementation            # Реализация камер, менеджеров рендеринга и вывода на экран
├── Interfaces                # Контракты для слабой связанности (ICamera, IScreen и др.)
├── Shape                     # Геометрические примитивы (Сфера, Треугольник, 3D-Объект)
├── StaticClass               # Вспомогательные утилиты (Загрузчик .obj, Игровое время)
├── Structure                 # Базовые структуры (Векторы, Лучи, Данные рендеринга)
├── UI                        # Простая оверлейная система для отрисовки текста поверх кадра
├── Network                   # Сетевой движок (TCP менеджер, пакеты, сериализатор)
```

### Главные компоненты архитектуры

* **`Frame`**: Сердце игрового цикла. Запускает бесконечный цикл, обновляет физику/логику сцены (`Update()`), вызывает отрисовку кадра на экране (`RenderFrame()`), высчитывает FPS и DeltaTime.
* **`Screen` (ConsoleScreenAsync)**: Абстракция дисплея. Содержит буферы цвета и яркости. Метод `RenderFrame` параллельно заполняет эти буферы данными, переводя яркость в символы из градиентной строки, после чего выводит готовый массив в консоль единым блоком.
* **`Scene`**: Контейнер для игровых сущностей. Хранит ссылки на активную камеру, список источников света и список отрисовываемых объектов (`IDisplays`). Отвечает за расчет цвета конкретного пикселя по UV-координате с помощью `GetPixelData`.
* **`DisplaysManager`**: Менеджер пересечений. Ищет ближайший к камере объект, с которым пересекается луч (Ray), выпущенный из экрана.

## 📐 Как устроен рендеринг (Pipeline)

Каждый кадр движок проходит следующие этапы вычислений:

1. **Генерация луча (Ray Generation)**:
   Камера преобразует UV-координату экрана в вектор направления луча в мировом пространстве с учетом углов поворота камеры (Pitch, Yaw, Roll).
2. **Проверка пересечений (Raycast)**:
   Луч передается в `DisplayManager`. Для каждого объекта на сцене вычисляется точка пересечения:
    * **Для сфер**: Аналитическое решение квадратного уравнения пересечения луча и сферы.
    * **Для 3D-моделей**: Сначала проверяется пересечение с Bounding Sphere (для отсечения заведомо невидимых объектов). Если луч попал в сферу, запускается цикл по всем треугольникам модели (алгоритм Моллера — Трамбора).
3. **Расчет освещения (Shading)**:
   Если пересечение найдено, рассчитывается освещенность в этой точке:
    * Вычисляется вектор направления на источник света.
    * Определяется косинус угла между нормалью поверхности и направлением света (диффузная составляющая).
    * Пускается теневой луч (**Shadow Ray**) из точки пересечения к источнику света. Если на пути есть преграда — точка считается затененной (яркость = 0).
4. **Буферизация и вывод**:
   Итоговая яркость масштабируется под размер градиентной строки символов. Символы записываются в буфер и выводятся на экран.

## ⌨️ Кроссплатформенная архитектура ввода

Движок использует слабосвязанный паттерн **Стратегия (Strategy)** для асинхронного неблокирующего ввода с аппаратной проверкой фокуса активного окна ОС. Это предотвращает «утечку фонового ввода» (background input leakage), гарантируя, что игра реагирует на клавиши только тогда, когда окно вашего терминала находится в фокусе.

```
[Input (Facade)] ---> [IInputProvider]
                            |
    +-----------------------+-----------------------+
    |                       |                       |                       
[User32 (Windows)]   [LibX11 (Linux)]        [DotNet (Fallback)]
```

*   **Windows (`User32InputProvider`)**:
    *   Прямой аппаратный опрос клавиш через системный вызов `GetAsyncKeyState`.
    *   Проверка фокуса активного окна: `GetForegroundWindow() == GetActualConsoleWindow()`. Нативно определяет связь «родитель-владелец» (`GetWindow` с флагом `GW_OWNER`), обеспечивая корректную работу как в классической консоли `conhost.exe`, так и в современном вкладочном **Windows Terminal** на Windows 11.
*   **Linux (`LibX11InputProvider`)**:
    *   Прямой аппаратный опрос клавиатуры через `XQueryKeymap` (считывание полной 256-битной карты состояния клавиш за один системный вызов раз за кадр).
    *   Динамический маппинг независимо от раскладки: преобразует стандартные значения `.NET` `ConsoleKey` в системные X11 KeySyms, которые при старте динамически переводятся в физические KeyCode под текущую активную раскладку игрока (QWERTY, AZERTY, йцукен и т.д.) через функцию `XKeysymToKeycode`.
    *   Проверка фокуса через обход дерева окон X11 вверх по иерархии, сопоставляя ID активного окна с переменной окружения `WINDOWID` терминала или отслеживая родительские имена процессов терминала.
*   **Универсальный фоллбек (`DotNetInputProvider`)**:
    *   Потокобезопасная асинхронная очередь ввода, оборачивающая стандартный метод `.NET` `Console.ReadKey(true)`.
    *   Использует **эмулятор отпускания клавиш на основе таймаута**, позволяя имитировать плавные события KeyUp и одновременные зажатия клавиш в условиях стандартного текстового потока ввода терминала.
    *   Автоматически активируется при подключении к серверам по SSH, в чистых сессиях Wayland или внутри изолированных контейнеров.

## 🌐 Сетевой протокол (Custom TCP Engine)

Сетевая подсистема написана с нуля на сокетах (`TcpListener`/`TcpClient`) без внешних зависимостей.

* **Формат пакета в сети**:
  При отправке формируется непрерывный массив байт следующей структуры (заголовок 12 байт + тело):
  ```
  [ Type ID (4 bytes) ] [ Sender ID (4 bytes) ] [ Payload Length (4 bytes) ] [  Payload Data (N bytes)  ]
  ```
* **Регистрация и сериализация (`PacketManager`)**:
  Пакеты наследуют интерфейс `INetworkPacket` и реализуют методы ручной записи/чтения в поток (`BinaryWriter`/`BinaryReader`). Каждому пакету присваивается уникальный ID на основе стабильного хэша от имени класса.
* **Событийная модель**:
  Клиентские скрипты подписываются на обработку конкретных типов пакетов через `PacketManager.Subscribe<T>((packet, senderId) => { ... })`.

## 🛠️ Как запустить и протестировать проект

Для сборки проекта вам понадобится **.NET 8.0 SDK** или новее.

### Шаг 1. Клонирование и структура файлов
Убедитесь, что файлы проекта разложены по папкам, указанным в исходном коде, а в корневой директории запуска исполняемого файла находится тестовая 3D-модель в формате `.obj` (например, стандартная низкополигональная обезьянка Blender — `monkey.obj`).

### Шаг 2. Компиляция и запуск
Перейдите в папку с проектом и выполните команду:
```bash
dotnet run --configuration Release
```
*(Использование конфигурации `Release` критически важно для работы `Parallel.For` на максимальной скорости)*.

### Шаг 3. Сетевой режим
При запуске программа предложит выбрать режим работы:
1. Выберите **`S`** (Server), чтобы запустить сервер. Программа покажет ваш локальный IP-адрес и порт.
2. На другом компьютере (или во втором терминале) запустите проект, выберите **`C`** (Client), введите IP-адрес сервера и подключитесь.

### Управление по умолчанию:
* **`W, A, S, D`** — Перемещение камеры (вперед, влево, назад, вправо).
* **`Space` / `Left Shift`** — Вверх / Вниз.
* **Стрелки клавиатуры** — Поворот камеры (осмотр во все стороны).
* **`Ctrl` + `W, A, S, D, Space, Shift`** — Управление положением источника света.
* **`+` / `-`** — Изменение мощности источника света.
* **`T`** — Открыть чат в сетевом режиме (ввод текста -> `Enter` для отправки, `Esc` для отмены).

## 🐧 Запуск нативного высокопроизводительного ввода на Linux

Чтобы включить опрос клавиатуры на аппаратном уровне ядра с нулевой задержкой и плавной обработкой одновременных нажатий клавиш (например, стрейф по диагонали через `W+A`), движок автоматически пытается запустить нативный ввод (`LibX11`).

Выполните следующие шаги, чтобы гарантировать нативную производительность в Unix-системах:

### В Linux (Обход Wayland vs. X11)
Современные дистрибутивы Linux (такие как Ubuntu 22.04+ или Fedora) по умолчанию работают под управлением графического сервера **Wayland**. В целях безопасности Wayland изолирует оконные сессии, полностью блокируя глобальный опрос ввода X11. Движок автоматически определяет активную сессию Wayland и переключается на безопасный буфер консоли (`DotNetInputProvider`).

Чтобы принудительно запустить высокопроизводительный опрос `LibX11` в системе с Wayland:
1. **Ловушка фонового демона (Terminal Server)**: Стандартный GNOME Terminal работает через единый фоновый процесс-демон. Обычные команды обхода Wayland игнорируются этим фоновым сервером, который продолжает оставаться в сессии Wayland.
2. **Решение**: Используйте независимый эмулятор терминала (например, `xterm`), принудительно запущенный в режиме совместимости с X11 (через XWayland):
   ```bash
   sudo apt install xterm
   WAYLAND_DISPLAY= xterm
   ```
3. В открывшемся окне `xterm` перейдите в папку вашего проекта и запустите игру через `dotnet run`. Движок успешно подключит аппаратный ввод через `LibX11InputProvider`.

*\*Примечание по контейнерам и Root:\* Если вы запускаете игру внутри Docker-контейнера или через `sudo`, перед запуском необходимо разрешить доступ к X11 на хост-машине:*
```bash
xhost +local:root
```

## 📝 Пример создания собственной сцены

Сцены создаются путем наследования базового класса `Scene`. Ниже представлен простейший пример сцены с одной сферой и одним источником света:

```csharp
using _3dEngine;
using _3dEngine.AbstractClass;
using _3dEngine.Implementation;
using _3dEngine.Interfaces;
using _3dEngine.Shape;

public class MyCustomScene : Scene
{
    private Camera _camera;
    private Sphere _sphere;
    private Light _light;

    public MyCustomScene(IDisplaysManagerAsync manager) : base(manager) { }

    public override void Start()
    {
        // Создаем камеру
        _camera = new Camera(new Vector3(0, 0, -5), Vector3.Zero);
        SetMainCamera(_camera);

        // Создаем красную сферу в центре координат
        _sphere = new Sphere(Vector3.Zero, Vector3.Zero, r: 1.5f);
        _sphere.Color = ConsoleColor.Red;
        AddDisplaysObject(_sphere);

        // Добавляем источник света
        _light = new Light(new Vector3(-2, 3, -4), lightPower: 15f);
        AddLight(_light);
    }

    public override void Update()
    {
        // Вращаем или перемещаем объекты каждый кадр, используя GameTime.GetDeltaTime()
    }
}
```
